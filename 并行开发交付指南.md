# 诡谜局 · 并行开发交付指南（前后端分工版）

> 传统前后端分离模式：Trae 负责前端，CodeBuddy 负责后端，通过接口契约解耦。

---

## 一、为什么前后端分工更高效？

| 对比维度 | 按模块分工 | **按前后端分工（✅推荐）** |
|----------|-----------|---------------------------|
| 上下文切换 | 高，每个模块横跨前后端 | **低**，各管一层 |
| 接口沟通成本 | 中，模块间接口多 | **低**，只有云函数调用一个接口层 |
| 联调难度 | 高，两边都可能改到对方的东西 | **低**，前端调后端 API，边界清晰 |
| 并行度 | 中，有依赖关系 | **高**，前后端完全解耦，可以同时开工 |
| AI 编写效率 | 中，AI 需要理解全栈 | **高**，AI 只需专注一层 |
| 合并冲突 | 多，可能改到同一个文件 | **少**，文件天然隔离 |
| **适合场景** | 微服务/独立功能模块 | **前后端分明的项目（本项目就是）** |

**关键原因**：本项目天然前后端分离 — 前端是静态 HTML+JS，后端是云函数，中间只有 `app.callFunction()` 一个调用方式。

## 三、接口契约（唯一共享文件）

前后端唯一需要对齐的文件是 `v2-modules/shared/api-contract.js`，它定义了：
- 每个云函数的名称、入参、出参
- 所有数据结构

Trae 前端不需要知道云函数内部实现，只需要按契约调用；CodeBuddy 后端不需要知道 UI 长什么样，只需要按契约返回数据。

### 云函数接口一览

```
前端调用方式（Trae 用）              后端实现（CodeBuddy 做）
─────────────────────────────────    ─────────────────────
app.callFunction({                  cloudfunctions/roomManager/
  name: 'roomManager',              ├── createRoom
  data: { action, ... }             ├── joinRoom
})                                  ├── leaveRoom
                                    ├── startGame
                                    ├── submitQuestion
                                    └── submitGuess

app.callFunction({                  cloudfunctions/characterManager/
  name: 'characterManager',         ├── getMyCharacters
  data: { action, ... }             ├── equip
})                                  ├── gacha
                                    ├── purchase
                                    └── getShop

app.callFunction({                  cloudfunctions/askAi/  (升级)
  name: 'askAi',                    └── 增加多人上下文参数
  data: { question, riddle, ... }
})

app.callFunction({                  cloudfunctions/handlePayment/  (升级)
  name: 'handlePayment',            └── 支持多商品
  data: { action, productId, ... }
})

```

## 五、前端任务详解（Trae）

### 5.1 总览

Trae 需要从零构建 V2 的前端，包含 5 个页面 + 核心逻辑。

**技术约束**：
- 纯原生 HTML+CSS+JS（与 V1 一致，不引入框架）
- 复用 V1 的 CSS 变量和暗黑中式恐怖风格
- CloudBase SDK 初始化方式见 `AGENTS.md` 和 `game.js`
- 所有后端调用统一通过 `api-contract.js` 定义的接口

### 5.2 页面清单

| 页面 | 文件 | 说明 | 优先级 |
|------|------|------|--------|
| 大厅 | `lobby.html` | 创建房间 / 输入房间号加入 / 开放房间大厅 | P0 |
| 房间游戏 | `room.html` | 谜题展示 + AI 对话 + 角色立绘 + 推理板 | P0 |
| 角色 | `character.html` | 角色展示 / 换装 / 抽卡动画 | P1 |
| 商店 | `shop.html` | 商品列表 / 购买 | P1 |
| 结算 | `result.html` | 排行榜 / MVP / 分享 | P1 |


## 八、常见问题

### Q: Trae 没有 CloudBase 环境怎么办？
A: 不需要。Trae 前端用 mock 数据开发，所有 `app.callFunction()` 调用先替换为 `MockAPI.xxx()`。联调时 CodeBuddy 部署好云函数后，Trae 只需改一行代码把 mock 换成真实调用。

### Q: 角色图片资源怎么处理？
A: 初期用 emoji 或纯色占位。正式版用 AI 生成原创角色图。Trae 前端只需要预留 `previewUrl` 字段展示图片即可。

### Q: 小程序和 H5 能共享代码吗？
A: 核心 JS 逻辑（调用云函数的部分）可以共享。UI 层需要分别开发。建议先做 H5。

### Q: 如何防止两人改到同一个文件？
A: 前后端文件完全分离：
- Trae 只改 `v2-frontend/` 目录下的文件
- CodeBuddy 只改 `cloudfunctions/` 目录下的文件
- 唯一的共享文件是 `v2-modules/shared/api-contract.js`，由 CodeBuddy 先写好，Trae 只读不写

### Q: 需要多久？
A: 前后端并行开发，预计 **3-4 周** 出 MVP（比模块分工的 4-6 周更快）。
