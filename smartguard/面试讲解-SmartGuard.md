> 用途：面试时张口就来的"讲得清"版本。原型已部署到 GitHub Pages（链接见末尾），本文件是配套讲解稿，所有数字/设备均来自 `prototype.html` 真实内容。
> 定位：面向工业设备的 AI 智能运维（AIOps）平台，作品目标是展示"全栈 + 行业理解"，投递 AI 训练 / 数据标注 / 政务数据辅助岗时的差异化亮点。

## 一、一句话定位

智维通 SmartGuard 是一个面向工业设备的 **AI 智能运维平台**：把分散在车间 / 场站的设备数据统一接入、实时监控、异常报警，并用"规则引擎 + 大模型"做故障诊断和预测性维护建议。原型内置两个行业场景：**新能源电站** 和 **汽车产线**，用来证明方案可跨行业复用。

| 维度 | 新能源电站 | 汽车产线 |
|------|-----------|----------|
| 设备总数（原型统计卡） | 48 台 | 36 台（3 条产线） |
| 典型设备 | 光伏阵列、逆变器、储能柜、充电桩、汇流箱、变压器、气象站 | 焊接机器人、冲压机、喷涂机器人、AGV 小车、拧紧机、视觉检测站 |

## 三、AI 诊断逻辑（面试核心，讲清"输入→处理→输出"）

**输入**：全场景设备的实时遥测 + 历史时序 + 当前报警记录 + 历史工单。

**处理（双引擎）**：
1. **规则引擎**（即时、可解释）：基于阈值 / 经验公式做硬判定。例：储能"电芯温差 >10℃"直接预警"BMS 均衡策略异常"；逆变器"输出波动 ≥5%"标记需检修。延迟低、结果可追溯，安全相关阈值不让模型单独决断。
2. **大模型层**（根因 + 预测）：把设备画像 + 异常序列 + 工单历史喂给大模型，输出根因推断（"怀疑 IGBT 模块老化，建议 48h 内检修"）和趋势预测（"送丝速度 120h 后或触发报警，建议保养时提前换送丝轮"）。

**输出**：① 全场景健康度评分；② 需关注设备清单（按严重度分级）；③ 分级处置建议（立即派单 / 限期内检修 / 持续监控）。

**兜底**：无 DeepSeek Key 时自动走规则版诊断，保证演示不空窗。

## 四、数据接入链路（Modbus / OPC UA / MQTT，讲清"设备怎么连上来"）

```
设备(PLC/控制器/传感器) ──协议采集──> 边缘网关 ──MQTT──> 云平台 ──> AI 诊断 ──> 报警/工单/大屏
        Modbus / OPC UA       协议归一/清洗        时序库        规则+大模型
```

1. **设备层采集（协议差异是关键知识点）**：
   - 新能源侧：逆变器 / 储能 BMS / 充电桩 / 气象站 多为 **Modbus RTU/TCP**（串口或以太网，工业现场最普遍、最便宜）。
   - 汽车产线侧：机器人 / 数控 / 压机控制器多原生支持 **OPC UA**（带信息模型、跨厂商互操作、自带安全鉴权，方便和 MES / SCADA 对接）。
   - 老设备只有 Modbus，新产线主推 OPC UA —— 所以**网关做协议融合**。
2. **边缘聚合**：边缘网关把多协议归一，做协议转换、数据清洗、断点续传（网络抖动时不丢数）。
3. **上云**：用 **MQTT**（轻量发布 / 订阅，适合车间不稳定网络、AGV 等移动设备）把时序数据推到云端 broker。
4. **云端闭环**：写入时序数据库（InfluxDB / TDengine）→ 触发规则引擎 + 大模型诊断 → 产出报警 / 工单 / 可视化大屏。

## 五、高频追问 & 应对（提前背熟）

- **Q：为什么用 MQTT 不用 HTTP？**
  A：设备侧网络差、数量大、要实时推送。MQTT 长连接 + 订阅更省带宽、断线可重连补发；HTTP 轮询开销大、难支撑万级设备。
- **Q：Modbus 和 OPC UA 怎么选？**
  A：简单仪表 / 老设备用 Modbus（简单便宜）；需要跨厂商互操作、带信息模型、安全鉴权用 OPC UA。网关做融合，平台侧不关心底层协议。
- **Q：大模型会不会误判导致误派单？**
  A：规则引擎兜底，安全相关阈值不让模型单独决断；模型只做辅助根因和建议，最终处置由运维确认。无 Key 也有规则版兜底。
- **Q：数据量多大、为什么用时序库？**
  A：单设备每秒若干点，万级设备日增亿级点位，关系库扛不住写入和聚合，所以用时序库（按时间分片、高压缩）。
- **Q：和市面产品（如树根、听云）差异？**
  A：轻量、双场景模板化、规则 + 大模型双引擎、可单机 / 私有化部署（SQLite 即可跑），适合中小场站快速落地。
