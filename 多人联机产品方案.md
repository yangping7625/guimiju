### 2.2 回合机制

| 模式 | 规则 | 适用场景 |
|------|------|---------|
| **轮流制** | 每人轮流提问，AI 逐一回答 | 标准模式，适合朋友聚会 |
| **抢答制** | 所有人同时打字，AI 回答第一个提交的问题 | 竞技模式，更快节奏 |
| **合作制** | 所有人可看到彼此问题，共同推理 | 新手友好，教学场景 |
| **竞技制** | 每人独立提问，互不可见，比谁先猜对 | 排位赛，PK 模式 |

### 2.3 胜利条件

- **单人猜对**：该玩家获胜，其他人可选择继续或查看答案
- **团队猜对**：合作模式下所有人获胜
- **超时**：设定时间上限（默认 15 分钟），超时则 AI 揭示答案

## 四、技术架构

### 4.1 整体架构

```
┌──────────────────────────────────────────────────┐
│                    前端 (H5/小程序)                 │
│  ┌──────────┐ ┌──────────┐ ┌──────────────────┐  │
│  │ 房间 UI   │ │ 角色渲染  │ │ 动画/特效系统    │  │
│  │ (Phaser)  │ │ (Live2D) │ │ (粒子+缓动)      │  │
│  └──────────┘ └──────────┘ └──────────────────┘  │
└──────────────────┬───────────────────────────────┘
                   │
    ┌──────────────┼──────────────┐
    │              │              │
    ▼              ▼              ▼
┌─────────┐  ┌──────────┐  ┌──────────┐
│ 云数据库 │  │ 实时推送  │  │ 云函数    │
│ (NoSQL) │  │ (Watch)  │  │ (API)    │
└─────────┘  └──────────┘  └──────────┘
    │              │              │
    └──────────────┼──────────────┘
                   │
                   ▼
            ┌──────────────┐
            │  DeepSeek API │
            │  (AI 主持人)   │
            └──────────────┘
```

### 4.2 技术选型

| 层级 | 技术 | 说明 |
|------|------|------|
| **前端框架** | 原生 JS + Phaser 3 | 当前 V1 已是原生 JS，多人版加入 Phaser 处理角色动画和场景 |
| **角色系统** | Spine / Live2D WebGL | 2D 骨骼动画，支持换装。初期可用静态图+CSS 动画代替 |
| **实时通信** | CloudBase 数据库 Watch | CloudBase NoSQL 的 `.watch()` 实现房间内实时同步 |
| **房间管理** | 云函数 + NoSQL | 创建/加入/离开房间，状态管理 |
| **AI 回答** | 现有 askAi 云函数（复用） | 增加多人上下文（谁问的、当前轮到谁） |
| **支付** | 现有 handlePayment（复用） | 增加多商品管理 |
| **部署** | CloudBase 静态托管 + 云函数 | 与 V1 完全一致 |

### 4.3 为什么用 CloudBase Watch 而不是 WebSocket？

1. **零额外成本**：CloudBase NoSQL 自带实时推送，不需要额外部署 WebSocket 服务
2. **与现有架构一致**：V1 已经用了 CloudBase，不需要引入新技术栈
3. **开发简单**：`.watch()` API 只需几行代码
4. **自动扩缩**：无需担心并发连接数

**限制**：Watch 的延迟约 100-300ms，对于回合制推理游戏完全够用。

### 4.4 数据库设计

```
📦 guimiju_rooms（房间表）
├── roomId: string (PK)
├── hostId: string (房主 openid)
├── puzzleId: number (当前谜题)
├── mode: 'turn' | 'race' | 'coop' | 'compete'
├── maxPlayers: number
├── status: 'waiting' | 'playing' | 'finished'
├── currentTurn: number (当前轮到第几个玩家)
├── players: [
│     { id, name, avatar, character, seat }
│   ]
├── chatLog: [
│     { playerId, question, answer, timestamp }
│   ]
├── createdAt: timestamp
└── updatedAt: timestamp

📦 guimiju_characters（角色表 - 用户拥有）
├── _openid: string
├── characterId: string
├── skinId: string
├── obtainedAt: timestamp
└── isEquipped: boolean

📦 guimiju_items（道具/皮肤表 - 用户拥有）
├── _openid: string
├── itemType: 'skin' | 'effect' | 'frame' | 'title' | 'room'
├── itemId: string
├── obtainedAt: timestamp
└── isEquipped: boolean

🔑 user_unlocks（兑换码解锁记录）
├── _openid: string
├── productId: string (如 'unlock_all_90')
└── unlockedAt: timestamp
```

### 5.2 参考图风格分析

你发的参考图核心特征：
- **二次元角色立绘**在底部横向排列
- **便签纸/公告板风格**的谜题展示
- **聊天室式**的对话区
- **角色名字标签**显示在立绘上方
- 整体偏明亮温馨（而非恐怖黑暗）

**诡谜局的差异化方向**：
- 保留暗色调但降低恐怖感 → **神秘推理风**（深紫+暗金，类似侦探事务所）
- 角色风格：日系二次元或 Q 版（而非写实恐怖）
- 房间主题多样化（中式古宅只是其中一个主题）

## 七、风险 & 注意事项

| 风险 | 影响 | 应对 |
|------|------|------|
| AI 成本上升 | 多人模式提问量 ×N | 限制每人提问次数（如 10 次），超限用提示 |
| Watch 延迟 | 回合切换不够实时 | 加本地乐观更新 + 服务端确认 |
| 角色版权 | 二次元形象侵权风险 | 使用 AI 生成原创角色 / 购买商用素材 |
| 付费转化 | 内购模式用户接受度 | 先用通行证+皮肤测试，再上抽卡 |
| 审核风险 | 小程序可能审核慢 | H5 先行，小程序后续跟进 |

## 八、并行开发分工

采用**前后端分离**模式，效率更高：

| 角色 | 负责 | 内容 |
|------|------|------|
| **Trae（前端）** | 全部前端 | 5 个页面（大厅/房间/角色/商店/结算）+ 动画/特效 + 状态管理 |
| **CodeBuddy（后端）** | 全部后端 | 5 个云函数 + 数据库 + 安全规则 + 部署 |

前后端通过 `v2-modules/shared/api-contract.js` 定义的云函数接口契约解耦。Trae 前端先用 mock 数据开发，联调时替换为真实调用。

详见 `并行开发交付指南.md`。
